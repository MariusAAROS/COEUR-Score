3.	System Features
3.1	START NEW PROJECT
3.1.1	Description and Priority
The START NEW PROJECT function shall allow the user to create a new project.
3.1.2	Functional Requirements
3.1.2.1	The user shall be prompted to provide the new project name.
3.1.2.2	The user shall be prompted to provide the low level document path.
3.1.2.3	The user shall be prompted to provide the high level document.
3.1.2.4	The user shall optionally be prompted to select an IR Method from the following list:
•	Vector-Space Retrieval
•	Keyword Extraction
•	Keyword Extraction-IDF
3.1.2.5	The user shall optionally be prompted to select a Vocabulary Base of either “Both Documents” or “Low Level Document Only”.  The default value is “Both Documents”.
3.1.2.6	The user shall optionally be prompted to provide Weight Option from the following list:
•	TF-IDF (Default)
•	Okapi
•	LTU
3.1.2.7	If the user has selected an IR Method of Keyword Extraction or Keyword Extraction-IDF, the user must also select a Weight Threshold.
3.1.2.8	The user shall optionally be prompted to provide a Thesaurus Filename.
3.1.2.9	The user shall optionally be prompted to provide the Stop Word file.
3.1.2.10	The user shall optionally be prompted to provide a Feedback Method.  “Standard Rochio” is currently the only option.
3.1.2.11	The user shall be able to cancel the process and return to the previous state at any time before the project is created.
3.1.2.12	The user shall be able to create the project using the above input.  The project shall be placed in “Trace” mode.
3.2	Load Project
3.2.1	Description and Priority
Since the majority of projects are not completed in a single setting, then it is critical to provide the user the option to load an existing project to continue working on the project.  Load Project feature will provide the user the capacity to open an existing project in the last saved state with all the associated mode options. [1]

3.2.2	Functional Requirements
3.2.2.1	The user shall be prompted to select a project name from a list of saved projects.  Project names are obtained by scanning the saved projects directory.
3.2.2.2	The user shall be prompted to select a feedback method.  The only option is “Standard Rochio”.
3.2.2.3	The user shall be able to cancel the LOAD PROJECT action and return to the previous activity.
3.2.2.4	The user shall be able to load the project once the project name and feedback method is selected.
3.3	LOAD RTM
3.3.1	Description and Priority
Similar in functionality to the START NEW PROJECT function, the LOAD RTM function collects project parameters from the user and initializes the project.  A project created via the LOAD RTM function may be in either “trace” or “assess” mode.  Assess mode is usually used when the user is provided a RTM and needs to evaluate its accuracy, completeness, and satisfaction level.
3.3.2	Functional Requirements
3.3.2.1	The user shall be prompted to provide the new project name.
3.3.2.2	The user shall be prompted to provide the low level document path.
3.3.2.3	The user shall be prompted to provide the high level document path.
3.3.2.4	The user shall be prompted to provide the RTM file path.
3.3.2.5	The user shall optionally be prompted to select an IR Method from the following list:
•	Vector-Space Retrieval
•	Keyword Extraction
•	Keyword Extraction-IDF
3.3.2.6	The user shall optionally be prompted to select a Vocabulary Base of either “Both Documents” or “Low Level Document Only”.  The default value is “Both Documents”.
3.3.2.7	The user shall optionally be prompted to provide “Weight Option” from the following list:
•	TF-IDF (Default)
•	Okapi
•	LTU
3.3.2.8	If the user has selected an IR Method of “Keyword Extraction” or “Keyword Extraction-IDF”, the user must also select a “Weight Threshold”.
3.3.2.9	The user shall optionally be prompted to provide a “Thesaurus” filename.
3.3.2.10	The user shall optionally be prompted to provide the “Stop Word” filename.
3.3.2.11	The user shall optionally be prompted to provide a “Feedback Method”.  “Standard Rochio” is currently the only option.
3.3.2.12	The user shall optionally be prompted to provide the “Project Mode” of either “Trace” or “Assess”.  “Trace” is the default mode.
3.3.2.13	The user shall be able to cancel the process and return to the previous state at any time before the project is created.
3.3.2.14	The user shall be able to create the project using the above input.  The project shall be placed in specified “Project Mode”.
3.4	SAVE
3.4.1	Description and Priority
The user shall be able to activate the SAVE function to save the current data and status of the project.
3.4.2	Functional Requirements
3.4.2.1	When activated RETRO will save all current data and project.
3.5	CLOSE CURRENT PROJECT
3.5.1	Description and Priority
The CLOSE CURRENT PROJECT Function will allow the user to close a given project without exiting the RETRO application.
3.5.2	Functional Requirements
3.5.2.1	The user shall be able close the current project.
3.6	EXIT
3.6.1	Description and Priority
It is necessary to provide a mechanism to exit the RETRO application when the user is finished working with the application.
3.6.2	Functional Requirements
3.6.2.1	The user shall be able to exit the application.
3.7	SHOW XML
3.7.1	Description and Priority
The user may want to view the current project’s XML file format.  The SHOW XML function will provide the user a display of the current XML file.
3.7.2	Functional Requirements
3.7.2.1	The user shall be able to view the XML version of the RTM.
3.8	GENERATE REPORT
3.8.1	Description and Priority
After a project has completed either a trace or an assess process, then the user will need the capability to review the final results of the process in an easy to read format other than the XML file format that is automatically generated throughout the process.  The GENERATE REPORT function will provide the user a list of reports that the user may view.  The user will have the option to save the generated report or modify the design of the report. [1]
3.8.2	Functional Requirements
3.8.2.1	The user shall be able to generate and save the “RTM report”
3.8.2.2	The user shall be able to generate and save the “Parent with no children report”
3.8.2.3	The user shall be able to generate and save the “Parents with no children with text report”
3.8.2.4	The user shall be able to generate and save the “Children with no parents report”
3.8.2.5	The user shall be able to generate and save the “Children with no parents with text report”
3.8.2.6	The user shall be able to generate and save the “Not satisfied requirements report”
3.8.2.7	The user shall be able to generate and save the “Not satisfied requirements with text report”
3.9	FILTER LOW LEVEL REQUIREMENTS
3.9.1	Description and Priority
“The user shall be able to limit the display of low-level requirements or candidate requirement links.” [1]
3.9.2	Functional Requirements
3.9.2.1	The user shall be able to designate whether the filter process applies to the selected high level document element (LOCAL) or to all high level document elements (GLOBAL).
3.9.2.2	The user shall be able to filter the list of low-level requirements by setting a cutoff point for the weight between 0 and 1.0.
3.9.2.3	The user shall be able to designate that the top N elements to be displayed.
3.9.2.4	The user shall be able activate a KILL HIDDEN Links function to remove all elements not currently displayed.
3.10	DISPLAY LOW LEVEL ELEMENTS
3.10.1	Description and Priority
The user will need the ability to view the text of low-level requirements.  The DISPLAY function will provide three different sort options to view the text of low-level requirements.  The user can sort the low-level requirements based upon the weight of linking relationships, original document order, or by requirement id. [1]
3.10.2	Functional Requirements
3.10.2.1	The user shall be able to sort the low level documents by “Requirement ID”
3.10.2.2	The user shall be able to sort the low level documents by “Weights”
3.10.2.3	The user shall be able to sort the low level documents by “Document Order”
3.11	FEEDBACK
3.11.1	Description and Priority
Since RETRO’s tracing and assessing modes are based upon algorithms that will allow processes to be improved as the system receives FEEDBACK from the user.
3.11.2	Functional Requirements
3.11.2.1	The user shall select a low level requirement and identify the link status from the following list:
•	Link
•	No Link
•	Default
3.12	MULTIPLE PROJECTS
3.12.1	Description and Priority
The user shall be able to load multiple projects at the same time.
3.12.2	Functional Requirements
3.12.2.1	The RETRO System shall allow multiple projects to be open at the same time

3.13	IMPORT
3.13.1	Description and Priority
The user shall be able to import projects.
3.13.2	Functional Requirements
3.13.2.1	The RETRO System shall provide a mechanism to convert existing input text files into RETRO file format.  (Not implemented.)
3.14	GUI INTERFACE
3.14.1	Description and Priority
The user shall use a GUI INTERFACE to work with RETRO.
3.14.2	Functional Requirements
3.14.2.1	The RETRO System shall provide a GUI mechanism for the user to interact with the RETRO System.

4.	BROWSE MODE
4.1	KEYWORD SEARCH
4.1.1	Description and Priority
The user is provided the capability via BROWSE mode to manually link low-level requirements to high-level requirement. In order to help the user to quickly locate the desired low-level requirement or high-level requirement, a KEYWORD SEARCH function is offered to the user. [1]
4.1.2	Functional Requirements
4.1.2.1	The user shall be able to enter a keyword into a search text box to search within low-level requirements and the system shall return matches.
4.1.2.2	The user shall be able to enter a keyword into a search text box to search within high-level requirements and the system shall return matches.
4.2	MANUAL LINKING/TRACING
4.2.1	Description and Priority
There will be times when the user will not agree with RETRO’s list of candidate links.  Therefore, it is important to provide a mechanism for the user to manually change, create, or delete links between low and high level requirements.  BROWSE mode is the mechanism within RETRO that enables the user to perform MANUAL LINKING/TRACING. [1]
4.2.2	Functional Requirements
4.2.2.1	The user shall be able to create and remove low-level requirements with high-level requirements.

5.	TRACE MODE
5.1	TRACING
5.1.1	Description and Priority
One of the purposes of RETRO is to aid the user in linking low-level requirements with high-level requirements.  TRACING functionality is the component of RETRO that automatically generates a list of candidate links of high to low-level requirements.
5.1.2	Functional Requirements
5.1.2.1	RETRO shall provide a mechanism for the user to generate a trace on all requirements.
5.1.2.2	RETRO shall provide a mechanism for the user to generate a trace on a given requirement(s).
5.2	TRACING RESULTS
5.2.1	Description and Priority
After the TRACING function is performed, then the results of the TRACING process will need to be made available to the user to view.  The TRACING RESULTS functionality will display a list of all candidate links of high to low-level requirements based on any given filtering criteria.  The user will be allowed to provide feedback on the TRACING RESULTS in order to improve future TRACING.
5.2.2	Functional Requirements
5.2.2.1	RETRO shall display the list of candidate links based on the tracing and permit feedback, filtering, and display functionality.
5.3	SAVE TRACE
5.3.1	Description and Priority
The SAVE TRACE function shall allow the user to save the trace result.
5.3.2	Functional Requirements
5.3.2.1	RETRO shall provide a mechanism to save the current status of trace.
5.4	COMPLETING TRACE
5.4.1	Description and Priority
The COMPLETING TRACE function shall allow user to finish the tracing process.
5.4.2	Functional Requirements
5.4.2.1	RETRO shall mark the project completed and not permit further changes.

6.	ASSESS MODE
6.1	RUN ACCURACY & COMPLETENESS
6.1.1	Description and Priority
The user needs a mechanism that will help to determine whether the TRACING was successfully performed based on accuracy and completeness.  An accuracy test tells the user that the candidate links provided are correctly related to the high-level requirement.  Completeness test tells the user that all possible links that are listed in the candidate list are included and no low-level requirements are missing. [1]
6.1.2   Functional Requirements
6.1.2.1	RETRO shall provide a mechanism for the user to be able to run an ACCURACY & COMPLETENESS check by performing a full trace on the requirements and comparing the IR method results to the given RTM. [1]
6.2	SHOW RECOMMENDATIONS LIST
6.2.1	Description and Priority
“The [user] will need to have a list of recommendations based on RETRO’s TRACING process.  SHOW RECOMMENDATION will display to the [user] a list of recommendation while in ASSESS mode.  The [user] will have the capability to provide FEEDBACK to the RETRO system on whether or not the [user] agrees with RETRO’s assessment of the linking between the low-level requirements to the high-level requirements.” [1]
6.2.2	Functional Requirements
6.2.2.1	RETRO shall display/hide recommendations at the user’s request.
6.3	RUN SATISFACTION ANALYSIS
6.3.1	Description and Priority
Satisfaction function shows how satisfied the high-level requirement is with the given set of low-level requirements. [4] The Satisfaction Analysis function is checking the level of percentage for a satisfaction of the linked result with the high and low level input files. Once final output is agreed, the list of high level requirement is updated with options like “Satisfied”, “Not Satisfied” and “Partially Satisfied”. The status of high level requirement is color coded. Unsatisfied result can be repeated until all the high levels has been satisfied stage.
6.3.2	Functional Requirements
6.3.2.1	RETRO shall display to the user the percentage level of link. [1]
6.4	REMOVE ALL NO LINKS
6.4.1	Description and Priority
While in ASSESS mode, the user will need a mechanism to allow the removal of all low-level requirements that are not linked to a high-level requirement.  REMOVE ALL NO LINKS function will provide an easy way for the user to quickly remove low-level requirements with status of “No Link”.
6.4.2	Functional Requirements
6.4.2.1	RETRO shall remove all links marked as “No Link”
6.5	DELETE RECOMMENDED LINKS
6.5.1	Description and Priority
While in ASSESS mode, the user will need a mechanism that will allow the deletion of recommended links that the user feels are not correctly linked.
6.5.2	Functional Requirements
6.5.2.1	RETRO shall permit the user to select a recommended link and delete it.
6.6	COLOR-CODED SATISFACTION STATUS
6.6.1	Description and Priority
“In order to help the user to quickly identify the level of satisfaction of each candidate link, RETRO will need to provide a visual aid.  COLOR-CODED SATISFACTION STATUS function will color code each candidate link based on its satisfaction level.   The various colors will be a visual aid to the user to help quickly identify each links satisfaction level.” [1]
6.6.2	Functional Requirements
6.6.2.1	RETRO shall COLOR-CODE THE SATISFACTION STATUS of each given candidate link.
